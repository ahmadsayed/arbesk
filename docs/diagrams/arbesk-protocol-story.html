The Arbesk Protocol Story

A content-addressed pipeline for 3D assets: upload, decompose, deduplicate, version, and download — where unchanged bytes are never stored twice.

Scroll ↓

1. Upload & Decompose

An uploaded asset is split into textures, mesh, and animation. Each part is hashed and pushed to IPFS independently.

Textures Mesh Animation Stored Reused CID
Content addressing means identical blocks across branches collapse into a single CID.

2. Download & Recompose

A request fetches blocks by CID, resolves duplicates, verifies hashes, and reassembles the asset.

Textures Mesh Animation Fetched block Reused CID
Download is the mirror of upload: blocks are verified by hash, then composed back into the original asset.

3. Cross-Asset Deduplication

Multiple assets share common building blocks. IPFS stores each unique block only once, no matter how many assets reference it.

Texture block Mesh block Shared / reused block
Asset 1 and Asset 2 share one texture; Asset 2 and Asset 3 share one mesh. Storage grows with uniqueness, not with copies.

4. Micro-Ledger: Version Chain

Every new version is a manifest that references the previous one. Only changed content blocks are stored; the rest are reused by CID.

Manifest New / changed block Shared block (same CID)
The manifest chain is an immutable micro-ledger. Versions are cheap because unchanged blocks never move.
